# Agent 插件平台规划（概念阶段）

> 状态：💭 思路存档 / 待与开发者社区沟通  
> 创建时间：2024-12-10  
> 预计版本：v2.1 或更高

---

## 📌 背景与动机

在 v2.0 统一分析引擎上线后，我们观察到：

1. **资深开源用户已经在做二次开发**
   - 基于开源版本封装自己的行业模型、择时逻辑、因子策略
   - 有些用户已经开始搭建自己的平台分发给其他人

2. **潜在风险**
   - 如果不提供官方的合作路径，这些二开用户可能会分走我们的用户
   - 他们的能力和成果无法反哺到官方生态

3. **机会点**
   - 把这部分力量变成合作关系
   - 平台提供分发和变现渠道，开发者专注在分析能力上
   - 双赢：用户获得更丰富的分析能力，开发者获得收益

---

## 🎯 核心目标

> **核心引擎收口在我们手里，分析能力往外开放给开发者做"插件化扩展"，并且可在我们的平台上变现。**

## 🏗️ 架构设计思路

### 1. 插件接口定义（AgentPlugin Protocol）

```python
class AgentPlugin(Protocol):
    id: str                  # 唯一 ID
    version: str
    display_name: str
    description: str
    config_schema: dict      # JSON Schema，前端自动生成配置表单
    categories: list[str]    # 分类：["大盘分析", "行业分析", "情绪因子"]

    def run(self, inputs: dict, config: dict, context: RunContext) -> dict:
        """执行插件逻辑"""
        ...
```

- WorkflowEngine 内部不关心"怎么分析"，只负责**调度插件**
- 现有内置 Agent 也可以看作"内置插件"，和第三方插件统一接口

### 2. 插件注册机制（Plugin Registry）

- 启动时扫描 `plugins/` 目录、数据库、或 entrypoints
- 加载所有插件的 `manifest`：
  - 插件 ID / 版本 / 作者 / 授权协议
  - 所属分类（大盘分析、行业轮动、资金面、另类数据、择时因子…）
  - 依赖（需要哪些数据源 / LLM 能力）
  - 收费模式（免费 / 付费 / 试用）

### 3. 前端工作流编辑器集成

- 节点面板新增"来自插件市场"分类
- 用户选中插件节点时，右侧面板根据 `config_schema` 自动渲染配置 UI
- 类似低代码平台的组件属性面板

### 4. 执行层：统一的"沙箱调用"

**纯提示词型插件（安全、简单）：**
- 只提供 prompt 模板 + 聚合逻辑
- 由平台的 LLM 客户端统一调用

**代码型插件（需要沙箱隔离）：**
- 以"远程 HTTP 服务"或"容器化 Runtime"方式接入
- 通过安全网关调用，核心引擎只和稳定的 HTTP/Tool 协议打交道
- 避免第三方代码直接跑在主进程里

## 📊 实现难度与时间评估

| 阶段 | 内容 | 难度 | 预估时间 |
|------|------|------|----------|
| **MVP 版** | 纯提示词型插件 + 简单市场页 + 手动审核 | ⭐⭐ 中等 | 3-4 周 |
| **基础版** | 配置化插件 + 自动安装/卸载 + 前端动态渲染 | ⭐⭐⭐ 中高 | 6-8 周 |
| **完整版** | 代码型插件 + 沙箱执行 + 支付结算 + 开发者后台 | ⭐⭐⭐⭐ 高 | 3-4 个月 |

### 各模块工作量拆解

**后端插件系统（核心）：**
- 插件接口定义：1-2 天
- 插件注册表 + 加载器：3-5 天
- 与 WorkflowEngine 集成：2-3 天
- 生命周期管理（安装/启用/禁用）：3-5 天

**插件市场后端：**
- 元数据管理 + API：3-5 天
- 审核流程：2-3 天
- 付费/订阅系统：5-10 天（涉及合规）
- 开发者结算：3-5 天

**前端：**
- 插件市场页面 + 详情页：5-7 天
- 工作流编辑器动态加载插件节点：3-5 天
- config_schema 动态渲染配置面板：3-5 天

**开发者工具 & 文档：**
- 插件 SDK + 示例插件：3-5 天
- 开发文档：2-3 天
- 提交/审核流程：2-3 天

**安全 & 沙箱（代码型插件）：**
- 沙箱隔离方案设计：5-7 天
- 实现 + 测试：7-14 天
- 这块是最难的，如果只支持"提示词型插件"可以先绕过

## 🤝 商业模式思路

### 平台提供的价值

1. **稳定的"舞台"**：统一工作流引擎、前端编辑器、运行平台与用户基础
2. **分发渠道 & 变现基础设施**：插件市场、支付结算、生命周期管理
3. **开放 API + SDK + 示例插件**：降低开发者接入门槛

### 对二开用户的引导策略

- 不建议 fork 一套平台、重新拉用户
- 把行业模型 / 因子 / 工作流封装成插件，上架官方市场
- 平台负责运维和分发，开发者专注策略和分析逻辑
- 收入按比例分成（如 7:3 / 6:4）

## 🚀 建议的实施路径

### v2.0 阶段（当前）：预留扩展点

不做完整插件市场，但在架构上预留好扩展点：

1. **统一 Agent 接口**：把现有内置 Agent 抽象成统一的 Plugin 基类/协议
2. **Registry 模式**：WorkflowEngine 通过 Registry 查找 + 统一协议调用
3. **config_schema 支持**：节点配置存储结构支持 JSON Schema 描述

**预计工作量：约 1 周**

### v2.1 阶段：MVP 插件市场

**第一步：纯提示词型插件（3-4 周）**
- 开发者只提供：manifest + prompt 模板 + 参数定义
- 简单的市场页面 + 手动审核上架
- 跑通"开发者提交 → 审核 → 用户安装 → 工作流使用"闭环

**第二步：代码型插件 + 完整生态（2-3 个月）**
- 支持开发者提交 Python 代码
- 沙箱执行 / 容器化隔离
- 完整支付结算 + 开发者后台
- 自动化审核 + 版本管理

## ❓ 待确认问题

在正式启动前，需要和资深开发者沟通确认：

1. **他们现在在做什么？**
   - 封装了哪些能力？行业模型？择时策略？数据源对接？
   - 是自己用，还是想分发给其他人？

2. **他们的诉求是什么？**
   - 想要变现渠道？还是只是想贡献回社区？
   - 对分成比例有什么预期？

3. **技术上的偏好？**
   - 更愿意只提供提示词，还是想跑自己的代码？
   - 对沙箱隔离、审核流程有什么顾虑？

4. **合作意愿？**
   - 愿意把现有二开成果迁移到官方插件市场吗？
   - 有没有担心的点（如知识产权、收益分配）？
