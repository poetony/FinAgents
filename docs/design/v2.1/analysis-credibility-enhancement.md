## 🎯 问题背景

### 核心问题

大语言模型（LLM）的训练数据是静态的（截止到某个时间点，如2024年），但股票分析需要基于最新的实时数据。当前系统虽然可以通过工具调用获取最新数据，但在综合分析阶段，LLM 可能会混入训练数据中的过时信息。

### 问题场景

| 问题类型 | 具体场景 | 影响 |
|---------|---------|------|
| **混入过时数据** | 子报告说PE是25倍，但LLM"记得"以前是30倍，混淆使用 | 估值判断错误 |
| **编造信息** | 子报告没提到某个利好，但LLM编造"公司近期有重大合作" | 误导投资决策 |
| **错误推理** | 基于LLM训练时的行业格局做判断，但行业已经变化 | 行业分析失准 |
| **年份混乱** | 说"等待2024年年报"，但现在是2026年 | 时间线错误 |

## 🔍 解决方案分析

### 方案1：强化提示词约束

**描述**：在综合分析阶段的提示词中添加严格的数据约束。

**实现方式**：

```markdown
**严格数据约束**：
1. 你只能使用【上面提供的子报告】中的数据
2. 不要使用你训练数据中的任何"记忆"
3. 如果子报告中没有某个信息，就说"数据未提供"
4. 所有数据引用必须标注来源，例如"根据基本面报告..."
5. 当前分析日期是 {{analysis_date}}，不要提及已过时的年份
```

**优点**：
- ✅ 实现简单，只需修改提示词
- ✅ 无需改变系统架构
- ✅ 可快速部署

**缺点**：
- ❌ LLM 可能不完全遵守，特别是复杂推理时
- ❌ 无法事后验证是否遵守约束
- ❌ 依赖 LLM 的自觉性

**适用场景**：作为基础保障层，与其他方案配合使用

### 方案2：结构化引用输出

**描述**：要求 LLM 输出时强制引用数据来源，便于事后校验。

**输出格式示例**：

```json
{
  "conclusion": "建议买入",
  "confidence": 0.72,
  "reasoning": [
    {
      "point": "估值较低",
      "evidence": "PE 25倍，低于行业平均30倍",
      "source": "基本面报告",
      "source_quote": "当前PE为25倍，处于历史低位..."
    },
    {
      "point": "技术面向好",
      "evidence": "MACD金叉，站上20日均线",
      "source": "市场报告",
      "source_quote": "技术指标显示MACD于1月10日形成金叉..."
    }
  ],
  "risks": [
    {
      "point": "行业竞争加剧",
      "evidence": "毛利率下降至37%",
      "source": "基本面报告",
      "source_quote": "受竞争影响，毛利率同比下降3个百分点..."
    }
  ]
}
```

**校验机制**：

```python
def validate_references(output: dict, source_reports: dict) -> ValidationResult:
    """校验输出中的引用是否真实存在于子报告中"""
    errors = []
    for item in output.get("reasoning", []):
        source = item.get("source")
        quote = item.get("source_quote")
        if source in source_reports:
            # 检查引用是否存在于原报告中
            if quote not in source_reports[source]:
                errors.append(f"引用不存在: {quote[:50]}...")
    return ValidationResult(valid=len(errors)==0, errors=errors)
```

**优点**：
- ✅ 可追溯，每个结论有明确来源
- ✅ 可验证，事后可以检查引用是否真实
- ✅ 提高透明度，用户可以看到分析依据

**缺点**：
- ❌ 输出格式复杂，需要调整前端展示
- ❌ 校验逻辑需要开发
- ❌ 可能增加 LLM 输出长度

**适用场景**：需要高可信度、可审计的分析场景

### 方案3：RAG 增强（检索增强生成）

**描述**：将子报告切分成段落，存入向量数据库，综合分析时通过语义检索获取相关信息。

**架构设计**：

```
┌─────────────────────────────────────────────────────────────────┐
│                        子报告生成                                 │
│  市场报告、基本面报告、新闻报告、社媒报告、大盘报告、行业报告        │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                      文档处理层                                   │
│  1. 文档切分（按段落/主题）                                        │
│  2. 向量化（Embedding）                                           │
│  3. 存入向量数据库                                                 │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                      向量数据库                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ ID | 内容                      | 来源      | 向量       │   │
│  │ 1  | PE 25倍，处于历史低位...   | 基本面报告 | [0.1,...]  │   │
│  │ 2  | MACD金叉，站上20日均线...  | 市场报告   | [0.2,...]  │   │
│  │ 3  | 智算中心一期已投产...      | 新闻报告   | [0.3,...]  │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                      综合分析阶段                                  │
│                                                                  │
│  分析问题："光环新网的估值水平如何？"                               │
│                     │                                            │
│                     ▼                                            │
│              向量检索（语义相似度）                                 │
│                     │                                            │
│                     ▼                                            │
│  检索结果：                                                       │
│    [1] 基本面报告："当前PE 25倍，处于历史低位..."（相似度0.92）     │
│    [2] 市场报告："股价15.89元，较高点回落40%..."（相似度0.85）      │
│                     │                                            │
│                     ▼                                            │
│  LLM 只基于检索结果回答（不使用训练数据"记忆"）                     │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**优点**：
- ✅ 强制基于检索结果，大幅减少幻觉
- ✅ 语义检索，找到最相关的信息
- ✅ 可扩展，未来可加入年报、研报等更多数据源

**缺点**：
- ❌ 增加系统复杂度，需要引入向量数据库
- ❌ 需要选择和部署 Embedding 模型
- ❌ 实现成本较高

**技术选型建议**：

| 组件 | 推荐方案 | 备选方案 |
|------|---------|---------|
| 向量数据库 | MongoDB Atlas Vector Search | Chroma、Milvus |
| Embedding 模型 | BGE-large-zh | text-embedding-3-small |
| 文档切分 | LangChain RecursiveCharacterTextSplitter | 自定义 |

**适用场景**：需要最高可信度、计划扩展知识库的场景

**事实提取 Agent 设计**：

```python
class FactExtractionAgent:
    """事实提取 Agent - 从子报告中提取结构化事实"""
    
    EXTRACTION_PROMPT = """
    你是一个事实提取专家。请从以下分析报告中提取关键事实数据。
    
    **要求**：
    1. 只提取明确的数据和事实，不做任何判断或推理
    2. 每个事实必须标注来源（来自哪个报告）
    3. 保持数据的原始表述，不要改写
    4. 按类别组织（估值、价格、财务、新闻、情绪、大盘等）
    
    **输入报告**：
    {{reports}}
    
    **输出格式**：
    ```json
    {
      "extraction_date": "{{analysis_date}}",
      "facts": {
        "valuation": [
          {"fact": "PE 25倍", "source": "基本面报告", "quote": "..."}
        ],
        "price": [...],
        "financial": [...],
        "news": [...],
        "sentiment": [...],
        "market": [...]
      }
    }
    ```
    """
```

**优点**：
- ✅ 事实和推理分离，逻辑清晰
- ✅ 事实提取可校验（检查是否真的来自子报告）
- ✅ 推理阶段有明确的数据边界
- ✅ 改动相对较小，不需要引入新基础设施

**缺点**：
- ❌ 增加一个处理步骤，可能增加延迟
- ❌ 事实提取的质量影响后续分析
- ❌ 需要设计好事实的分类体系

**适用场景**：平衡实现成本和效果的最佳选择

## 🎯 推荐方案

### 短期（立即实施）

**方案4（两阶段分析）+ 方案1（提示词约束）+ 方案2（结构化输出）**

1. **添加事实提取 Agent**
   - 在看涨/看跌研究员之前，新增"事实提取"节点
   - 从所有子报告中提取结构化事实
   - 后续分析只基于事实清单

2. **强化提示词约束**
   - 更新所有综合分析 Agent 的提示词
   - 添加数据边界声明和时间上下文

3. **结构化输出**
   - 要求输出中包含数据来源引用
   - 开发简单的校验逻辑

### 中期（1-2个月）

1. **事后校验系统**
   - 开发引用校验功能
   - 检测可能的幻觉
   - 生成可信度评分

2. **前端展示优化**
   - 显示数据来源
   - 显示分析时效性
   - 显示置信度指标

### 长期（3-6个月）

1. **RAG 系统**
   - 引入向量数据库
   - 支持年报、研报等非结构化数据检索
   - 构建知识库

2. **持续优化**
   - 收集用户反馈
   - 优化事实提取质量
   - 改进幻觉检测算法
